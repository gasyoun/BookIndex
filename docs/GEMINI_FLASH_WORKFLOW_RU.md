## 1) Роль Gemini Flash

Gemini Flash используется как быстрый рабочий контур для задач, где важны скорость, широкий контекст, мультимодальность, черновая классификация и итеративная подготовка изменений. Он не заменяет основной регламент Codex и не получает право самостоятельно публиковать результат.

Основная формула:

```text
Gemini Flash = быстрый исследователь / черновой редактор / проверяющий гипотез
Codex + локальные скрипты + GitHub Actions = финальный исполнительный контур
```

Подходит хорошо:

1. Быстрый разбор больших Markdown/JSON/TS-файлов.
2. Поиск повторяющихся UI/контентных проблем.
3. Черновая классификация редакторских очередей.
4. Предложение UX-вариантов и microcopy на русском языке.
5. Генерация списков smoke-проверок для конкретных hash-route.
6. Первичная оценка полноты описаний, стенограмм, ссылок и карточек.
7. Сравнение нескольких вариантов текста или структуры навигации.

Подходит ограниченно:

1. Генерация патчей для UI: допустима как черновик, но финальная правка проходит через Codex и локальную сборку.
2. Массовая нормализация данных: только через воспроизводимый скрипт или маленький batch с diff-review.
3. LLM-as-judge: только дополнительный сигнал после детерминированных проверок.

Не подходит:

1. Самостоятельный `commit`, `push`, деплой или закрытие issue.
2. Массовые изменения корпуса без точного списка файлов, правил и проверки.
3. Финальное решение о лингвистической корректности без источника.
4. Хранение секретов, токенов, учетных данных или приватных ключей в контексте.

## 2) Архитектурная схема

```text
Задача пользователя
  -> intake: цель, scope, риск, ожидаемый экран/данные
  -> context pack: только релевантные файлы, маршруты, правила, команды
  -> Gemini Flash: анализ, черновик, список рисков, предлагаемые проверки
  -> Codex: применяет патч или переписывает решение по паттернам репозитория
  -> verification: scripts + build + route smoke + diff-review
  -> handoff: что изменено, что проверено, что осталось рискованным
  -> publish: только по явному запросу или по регламенту задачи
```

Ключевой принцип: Gemini Flash не должен быть "магическим вторым мнением". Его вклад должен превращаться в проверяемый артефакт: список файлов, таблицу решений, черновой patch plan, набор smoke-route, issue draft или чеклист.

## 3) Контекстный пакет для Gemini Flash

Перед передачей задачи Gemini Flash нужно собрать короткий context pack. Он не должен содержать весь репозиторий.

Минимальный пакет:

```md
# Gemini Flash Context Pack

## Цель
...

## Scope
- Разрешено менять/анализировать: ...
- Не трогать: ...

## Релевантные файлы
- `path/to/file`
- `path/to/file`

## Релевантные маршруты
- `aaz-index.html#v4/...`

## Правила проекта
- Русский UI и issue-тексты.
- Не утяжелять header.
- Не дублировать хлебные крошки и навигацию.
- Проверки важнее уверенности модели.

## Ожидаемый результат
- Анализ / таблица / план патча / список smoke-проверок.

## Формат ответа
- Findings
- Proposed changes
- Checks
- Risks
```

Нельзя передавать:

1. Секреты и токены.
2. Нерелевантные большие каталоги.
3. Неограниченную просьбу "посмотри весь проект".
4. Старые договоренности из чата без фиксации в Markdown.

## 4) Уровни автономности

| Уровень | Тип задачи | Что может делать Gemini Flash | Что обязательно проверяет Codex |
|---|---|---|---|
| A0 | Read-only анализ | Читать context pack, давать выводы, находить риски | Сопоставить с реальными файлами |
| A1 | Документация | Черновик раздела, структура, чеклист | Markdown diff, `git diff --check` |
| A2 | Контентные правки | Предложить batch и правила нормализации | Малый diff, кодировка, валидаторы |
| A3 | UI/навигация | Предложить UX-решение и smoke-route | Сборка, визуальная проверка маршрута |
| A4 | Деплой/issue | Только подготовить текст | Выполняет человек/Codex по явному запросу |

По умолчанию Gemini Flash работает на A0-A1. Переход на A2-A3 допустим только если есть точный scope и проверяемые критерии готовности.

## 5) Рабочий цикл

### 5.1 Intake

Перед запуском формулируем:

1. Что должно измениться.
2. Где это видно пользователю.
3. Какие файлы или маршруты затронуты.
4. Какой риск: документация, контент, UI, данные, сборка, публикация.
5. Какая проверка докажет результат.

### 5.2 Select

В контекст попадают только:

1. Ближайший код или Markdown.
2. Нужные data-файлы.
3. Один релевантный фрагмент регламента.
4. Текущий diff, если задача продолжает уже начатую работу.

### 5.3 Draft

Gemini Flash возвращает не финальное решение, а рабочий артефакт:

1. Таблица проблем.
2. План патча.
3. Черновой текст.
4. Список маршрутов для проверки.
5. Риски и вопросы.

### 5.4 Apply

Codex применяет изменения локально с учетом паттернов проекта. Если Gemini Flash предложил патч, его нельзя применять вслепую: сначала сверить импорты, структуру данных, стиль UI и существующие проверки.

### 5.5 Verify

Минимум для документации:

```bash
git diff --check -- docs/GEMINI_FLASH_WORKFLOW_RU.md
```

Минимум для runtime/UI:

```bash
node scripts/bundle.js
python scripts/check_encoding.py
python scripts/validate_content.py
python scripts/build_aaz_index.py
```

Дополнительно для UI:

1. Открыть точный `aaz-index.html#...` маршрут.
2. Проверить desktop и mobile, если менялась навигация или layout.
3. Зафиксировать, что не появилось дублирование header/breadcrumbs/tabs.

### 5.6 Handoff

Финальный отчет должен содержать:

1. Что изменено.
2. Почему это соответствует разделу 10 регламента.
3. Какие команды прошли.
4. Какие маршруты проверены.
5. Что осталось только предположением или требует человеческого решения.

## 6) Типовые сценарии для BookIndex

### 6.1 Редакторские очереди

Gemini Flash:

1. Группирует элементы по типу проблемы.
2. Находит повторяющиеся формулировки.
3. Предлагает русские статусы и короткие описания.
4. Выделяет случаи, где нужен источник, а не языковая догадка.

Codex:

1. Меняет данные или код.
2. Запускает валидаторы.
3. Проверяет UI-маршрут очереди.

### 6.2 Навигация и header

Gemini Flash:

1. Ищет визуальную перегрузку.
2. Предлагает, что убрать, перенести или объединить.
3. Составляет smoke-list: первый уровень меню, вкладки, breadcrumbs, mobile.

Codex:

1. Правит компоненты.
2. Собирает `aaz-index.html`.
3. Проверяет маршруты.

### 6.3 Навигационная реформа по `docs/NAVIGATION_RETHINK_RU.md`

Этот сценарий нужен, когда Gemini Flash просят не просто оценить header, а подготовить архитектуру изменения верхней навигации. Источник решения: `docs/NAVIGATION_RETHINK_RU.md`.

Цель изменения:

1. Основным публичным входом становится `index.html`: страница знакомства с книгой, проектом и главными маршрутами.
2. `aaz-index.html#v4/home/home` остается основным рабочим интерфейсом приложения с новой иерархией `Главная / Указатели / Материалы / Аппарат / Инструменты / Практикум`.
3. Все существующие hash-route сохраняются; меняется навигационная группировка, а не смысл адресов.
4. `Корпус` не возвращается в первый уровень: он остается последним подпунктом в `Материалы`.

Архитектура изменения:

```text
route state
  -> index.html объясняет книгу и проект
  -> главная кнопка открывает aaz-index.html#v4/home/home
  -> SPA определяет текущий смысловой раздел
  -> показывает компактный первый уровень
  -> показывает локальный второй уровень выбранного раздела
  -> оставляет рабочий контент прежнего route
  -> сохраняет совместимость старых hash
```

Gemini Flash не обязан угадать точные текущие имена tabs/routes. Его задача - выдать проверяемую карту соответствий "старый route -> новый раздел -> локальный пункт" и отметить места, где нужен Codex-inspection.

Файловые границы для плана:

1. `v3_app.js` - route parsing, active section detection, click handlers, сохранение state.
2. `v3_template.html` - CSS/markup верхней навигации, если структура header живет в шаблоне.
3. `aaz-index.html` - только rebuild-артефакт после `npm run build`, не ручная правка.
4. `index.html` - главная публичная вводная страница; не превращать в дубль рабочего приложения, держать коротким и ведущим в рабочий маршрут.
5. `tests/e2e/smoke.spec.js` - smoke-критерии для desktop/mobile, route compatibility и отсутствия наложений.
6. `docs/NAVIGATION_RETHINK_RU.md` - источник продуктового решения.

Порядок внедрения:

1. Inventory: составить таблицу всех текущих top-level кнопок, tabs и hash-route.
2. Mapping: разложить их по шести новым разделам; спорные пункты пометить как `needs human decision`.
3. State: описать функцию `getNavSectionForRoute(route)` и fallback для неизвестного route.
4. Render: первый уровень всегда короткий; второй уровень рендерится только для активного раздела.
5. Landing: `index.html` становится главным знакомством с книгой и ведет в `aaz-index.html#v4/home/home`.
6. Cleanup: убрать счетчики и длинные labels из кнопок первого уровня; перенести их в summary внутри разделов.
7. Verification: дать route smoke-list и ожидания для каждого экрана.

Критерии готовности:

1. На desktop первый уровень помещается в одну строку без наложений.
2. На mobile первый уровень не создает горизонтальный overflow и не перекрывает поиск.
3. `Корпус` отсутствует в первом уровне и доступен внутри `Материалы`.
4. `Сводный указатель` доступен как основной пункт раздела `Указатели`.
5. Старые route вроде `#v4/all/list`, `#v4/lexicon/list`, `#v4/scholar/viz/module/viz01?century=21` продолжают открываться.
6. `index.html` не показывает placeholder: это главный публичный intro, но не рабочий SPA-интерфейс.
7. Нет дублирования "первый уровень + breadcrumbs + локальные tabs" там, где локальная навигация уже объясняет позицию пользователя.

### 6.4 Корпус и источники

Gemini Flash:

1. Сверяет описания на русском.
2. Находит англицизмы, дубли и тяжеловесные фразы.
3. Предлагает редакторские формулировки.

Codex:

1. Вносит минимальный diff.
2. Запускает content validation.
3. Не меняет факты без источника.

### 6.5 Визуализации

Gemini Flash:

1. Предлагает названия, легенды, пустые состояния.
2. Проверяет, нет ли дублирования навигационных строк.
3. Формирует список экранов для route smoke.

Codex:

1. Проверяет DOM/layout фактически.
2. Убеждается, что график не пустой и не перекрывает текст.
3. Собирает бандл.

## 8) Формат результата от Gemini Flash

Ответ считается пригодным, если в нем есть:

1. Конкретные файлы или маршруты.
2. Проверяемые утверждения.
3. Список рисков.
4. Отдельно помеченные предположения.
5. Предложенные проверки.

Ответ считается непригодным, если:

1. Он пересказывает задачу без действий.
2. Предлагает массовый rewrite без необходимости.
3. Ссылается на несуществующие файлы.
4. Требует отключить проверки.
5. Смешивает факты источника и редакторскую догадку.

## 9) Первые этапы внедрения

2. Есть ответ Gemini Flash в формате раздела 8.

### Этап 2: Повторяемые smoke/eval сценарии

Собрать список типовых маршрутов:

4. `aaz-index.html#v4/scholar/list`
5. `aaz-index.html#v4/scholar/viz/module/viz01?century=21`

Для каждого маршрута фиксировать ожидаемые свойства: нет лишнего header-текста, нет дублирующей навигации, меню не наслаивается, русские статусы, поиск не ломает layout.

### Этап 3: Контентные mini-evals

Создать маленькие проверочные наборы для задач, где Gemini Flash может помогать:

1. Русификация статусов.
2. Краткость header и описаний.
3. Кликабельность ссылок на страницы.
4. Отсутствие дублирующих карточек.
5. Полнота редакторской очереди.

Сначала эти evals должны быть Markdown-чеклистами. Автоматизация появляется только после того, как чеклист стабилен.

### Этап 4: Ограниченная автоматизация

Только после этапов 1-3 можно обсуждать API-интеграцию Gemini Flash. До этого достаточно ручного или полуавтоматического использования через context pack.

Требования к автоматизации:

1. Model id задается конфигом и проверяется перед запуском.
2. Запросы логируются без секретов.
3. Ответ сохраняется как артефакт проверки, а не применяется автоматически.
4. Любой write проходит через локальный diff-review.
5. Публикация остается отдельным ручным шагом.

## 10) Decision log

1. Gemini Flash берем как быстрый вспомогательный контур, а не как замену Codex.
2. Первичный фокус: документация, редакторские очереди, UX-аудит, smoke-checklists.
3. Для данных и UI Flash может предлагать, но не утверждать финальный diff.
4. Источником правды остаются репозиторий, валидаторы, GitHub Actions и опубликованный GitHub Pages маршрут.
5. Навигационную реформу вести через отдельный context pack: `docs/NAVIGATION_RETHINK_RU.md` + route mapping + smoke-list.
6. Следующее практическое действие: начать с одного ручного context pack для реальной задачи и сохранить результат как пример в issue или отдельном отчете.

# Codex Task: BookIndex — привести README к фактическому составу репозитория

Репозиторий: `gasyoun/BookIndex`

## Контекст

README в целом актуален: версия `v4.3`, статус проекта, ключевые файлы и основные разделы соответствуют текущему состоянию репозитория. Но таблица структуры и список проектной документации неполные.

### Что считать уже подтверждённым
- `README.md` уже корректно описывает `v4.3`, single-file SPA, `aaz-index.html`, `app_data.json`, `codex_instruction_v2.md`, `CODEX_WORKFLOW_RU.md`, `KIDS_GUIDE_RU.md`.
- В репозитории реально существуют:
  - `CODEX_VISUALIZATIONS_RU.md`
  - `CODEX_VIZ_INSTRUCTIONS_RU.md`
  - `ISSUE_PUBLISH_CHECKLIST_RU.md`
- Также реально существуют:
  - `scripts/viz/build-viz-cache.js`
  - `scripts/viz/build-viz-cache-worker.js`

## Задача

Обновить только документацию, в первую очередь `README.md`, чтобы она точнее отражала текущий состав репозитория.

## Что нужно сделать

### 1. Обновить раздел `Структура проекта`
Добавь в таблицу структуры проекта недостающие документы, которые уже есть в репозитории:

- `CODEX_VISUALIZATIONS_RU.md` — регламент/архитектура визуализаций
- `CODEX_VIZ_INSTRUCTIONS_RU.md` — инструкция по реализации VIZ-модулей
- `ISSUE_PUBLISH_CHECKLIST_RU.md` — чеклист публикации issues без мусора и битой кириллицы

### 2. Расширить блок про документацию
Если в README есть список ключевых документов в верхней части или отдельный блок документации, добавь туда ссылки на:
- `CODEX_VISUALIZATIONS_RU.md`
- `CODEX_VIZ_INSTRUCTIONS_RU.md`
- `ISSUE_PUBLISH_CHECKLIST_RU.md`

Формулировки должны быть короткими и в стиле текущего README.

### 3. Аккуратно отразить служебные/спринтовые документы
По данным аудита в репозитории также есть служебные markdown-файлы, например:
- `SPRINT_v4.1_2026-04-20.md`
- `LINKING_AUDIT_2026-04-19.md`
- `RELEASE_NOTES_2026-04-*.md`

Не нужно раздувать основной README длинным архивным списком. Выбери один из двух аккуратных вариантов:

#### Предпочтительный вариант
Добавь компактный подраздел вроде `Служебные документы и аудит`, где кратко перечислены:
- sprint boards
- audit notes
- release notes
- publish checklists

#### Допустимый вариант
Добавь в таблицу структуры одну-две строки об этих документах как о группе, без перегрузки README.

Важно: не превращай README в полный каталог всех исторических markdown-файлов.

### 4. Не менять то, что уже актуально
Не переписывай без необходимости:
- секцию `Что нового`
- описание `v4.3`
- список функциональности
- ссылки на GitHub Pages
- инструкции по запуску и проверкам

### 5. Не вносить кодовые изменения
В этом задании не трогай:
- `v3_app.js`
- `v3_template.html`
- `app_data.json`
- `aaz-index.html`
- тесты
- CI

Меняется только документация.

## Ограничения

- Не выдумывай файлов, которых нет.
- Не удаляй существующие корректные пункты README.
- Не меняй стиль README радикально: нужно минимальное, точное, редакторское обновление.
- Не утверждай, что какие-то файлы отсутствуют, если они уже упомянуты в README и/или существуют.
- Не понижай статус `scripts/viz/build-viz-cache.js` и `build-viz-cache-worker.js`: они присутствуют и могут оставаться в README.

## Предпочтительный результат

README после правки должен:
1. остаться компактным;
2. по-прежнему быть удобным для первого входа;
3. точнее отражать реальную документацию репозитория;
4. не скрывать существующие codex/viz/publish документы.

## Формат ответа Codex

1. Кратко перечисли, что именно обновлено в README.
2. Покажи diff/patch.
3. Отдельно перечисли, какие файлы документации были добавлены в README.

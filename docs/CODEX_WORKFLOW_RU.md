# Регламент Codex для BookIndex

Цель: зафиксировать единые правила оформления и публикации, чтобы не повторять устные договоренности.

## 1) Что обязательно делать в каждом цикле

1. Вносить изменения в код, данные или документацию только в согласованном scope.
2. Запускать минимальные проверки из раздела 5 или явно фиксировать блокер среды.
3. Пересобирать `aaz-index.html` после изменений в runtime, шаблоне, данных или VIZ-модулях.
4. Обновлять `README.md`, если изменились команды, статус, публичные сценарии или структура проекта.
5. Делать `commit` и `push` в `main`, если задача требует публикации результата.
6. Оформлять задачу в GitHub Issues по шаблону из раздела 2.
7. После публикации давать ссылку на demo с `?v=<short_sha>` и точным hash-маршрутом изменённого экрана.

## 2) Формат GitHub Issue (эталон: #32)

Язык: только русский (допустимы англоязычные технические термины и имена разделов UI).

### Заголовок

- Продуктовые задачи: `[v4/<раздел>] <краткое действие/результат>`
- Служебные задачи: `[служебное] <краткое описание>`
- Бэклог: `[Backlog][vN][PX] <краткое описание>`

### Тело issue (обязательная структура)

```md
Цель: ...

Критерии готовности:
- ...
- ...

Сделано:
- ...
- ...

Проверки:
- `команда` — результат
- `команда` — результат

Связанные коммиты:
- `<short_sha>` — что сделано

Статус: выполнено / в работе / блокер
```

### Комментарий при закрытии

```md
Исправлено в `<short_sha>` (`main`).

Проверки:
- ...
- ...
```

## 3) Labels: обязательная схема (эталон: #42)

У каждого issue должны быть все 4 группы:

1. `priority:*` (`priority:P0` / `priority:P1` / `priority:P2`)
2. `area:*` (`area:ui-ux`, `area:navigation`, `area:search`, `area:data`, `area:infra`, `area:analytics`, `area:offline`, `area:export`, ...)
3. `type:*` (`type:feature` или `type:refactor`)
4. `phase:*` (`phase:v2` / `phase:v3` / `phase:v4`)

Дополнительно можно ставить: `enhancement`, `accessibility`.

Нельзя оставлять:

- issue без labels;
- issue без любой из 4 обязательных групп;
- смешение `area:ux` и `area:ui-ux` (использовать только `area:ui-ux`).

## 4) Защита от битой кодировки и мусора в issues

Перед закрытием issue обязательно:

```bash
python scripts/issue_quality_guard.py --repo gasyoun/BookIndex --issues <номер> --strict-template
```

Периодически (или после серии задач) проверять весь репозиторий:

```bash
python scripts/issue_quality_guard.py --repo gasyoun/BookIndex --state all
```

Если обнаружены `???`, массовые `?` или mojibake:

1. Немедленно исправить title/body/comment через API.
2. Повторно прогнать guard.
3. Только после зеленой проверки закрывать issue.

## 8) UX/контент договоренности, которые нельзя забывать

1. Ссылки на страницы (`стр. N`) делать кликабельными везде, где технически возможно.
2. Не дублировать одинаковые фрагменты текста/блоки в карточках без необходимости.
3. Не допускать визуального мусора: лишние пробелы, «ломаные» переносы, артефакты ударения.
4. В описательных текстах не навязывать `ё`; оставлять `ё` только там, где это важно по источнику.
5. **Лингвистическая точность**: В русских заголовках и списках заглавная буква ставится только в начале фразы и в именах собственных (согласно нормам русского языка).
6. Любое изменение, о котором просил пользователь, отражать в GitHub (issue + при необходимости комментарий о закрытии).

## 9) Definition of Done (коротко)

Задача считается завершенной только если одновременно:

1. Код/контент изменен.
2. Проверки из раздела 5 выполнены (или явно зафиксирован блокер среды).
3. Изменения запушены в `main`.
4. Issue оформлен по разделам 2 и 3.
5. Пользователю дана актуальная ссылка на demo с `?v=<short_sha>`.

## 10) Применимость agent-roadmap-2026 к BookIndex

Источник: https://github.com/codejunkie99/agent-roadmap-2026

Оценка: подход применим к нам частично, но полезная часть высокая. Для BookIndex это не программа "строим отдельную agent-платформу за 17 недель", а дисциплина работы Codex с репозиторием: управляемый контекст, проверяемые инструменты, регрессионные проверки, понятные границы автономности и аккуратная публикация.

Что берем как рабочее правило:

1. **Context engineering вместо устных договоренностей.** Все устойчивые знания о проекте должны жить в репозитории: `docs/CODEX_WORKFLOW_RU.md`, `README.md`, issue, данные, скрипты проверок и маршруты demo. Важные договоренности нельзя оставлять только в чате.
2. **Harness как набор локальных правил и команд.** Наш harness - это Codex + файловая структура BookIndex + `scripts/*` + GitHub Actions + GitHub Pages. Новый LangGraph/Deep Agents слой не нужен, пока задача сводится к сайту, корпусу, визуализациям и публикации.
3. **Инструменты должны быть измеримыми.** Для каждой нетривиальной правки фиксировать, какие команды или маршруты реально проверяли результат. Для UI-изменений обязательно указывать точный hash-route, для данных - скрипт валидации и затронутый источник.
4. **Регрессионные проверки важнее "уверенности агента".** Минимальный набор из раздела 5 остается базой. Для рискованных задач добавлять точечные проверки: сборку, route smoke-test, поиск дубликатов, проверку кодировки, проверку конкретного JSON/TS-модуля или Pages-деплой.
5. **Контекст должен быть малым и релевантным.** Перед правкой читать ближайшие файлы и существующие паттерны, а не грузить весь проект. При длинных задачах фиксировать короткий рабочий summary: что менялось, какие команды прошли, что осталось рискованным.
6. **Автономность ступенчатая.** Читать файлы, искать по коду и запускать локальные проверки можно свободно. Менять файлы - только в scope задачи. `commit`, `push`, деплой, закрытие issue и массовые правки данных - только когда это явно нужно по задаче или прямо запрошено.

Что берем осторожно:

1. **Параллельные агенты и fan-out.** Полезны для независимых исследований, ревью разных областей или проверки разных маршрутов. Не применять к тесно связанным UI/данным, где легко получить конфликт или потерять общий контекст.
2. **LLM-as-judge и eval-подход.** Для BookIndex сначала нужны детерминированные проверки. LLM-оценку можно использовать как дополнительный слой для качества описаний, полноты стенограмм, редакторских очередей и подозрительных мест, но не как единственный критерий готовности.
3. **Долгая память агента.** Предпочтительный формат памяти - versioned Markdown/JSON в репозитории и GitHub Issues. Внешнюю память, vector store или agent database заводить только при доказанной проблеме поиска или повторяемости.

Что пока не берем:

1. Полноценный LangGraph/Deep Agents runtime для управления самим BookIndex.
2. Собственную платформу трассировки agent-траекторий, пока достаточно GitHub Actions, локальных логов и воспроизводимых команд.
3. Автоматические unattended-изменения в `main` без явного запроса пользователя и зеленых проверок.
4. Отдельные файлы `MY_ROADMAP.md` и `AGENTS.md` по протоколу внешнего репозитория: у нас уже есть проектный регламент, его и обновляем.

Практический вывод: применимость высокая для методики работы и контроля качества, средняя для multi-agent-паттернов, низкая для тяжелого production-agent стека. Следующий полезный шаг для нас - постепенно превращать повторяющиеся проверки BookIndex в явные smoke/eval сценарии, привязанные к маршрутам `aaz-index.html#...`, данным корпуса и GitHub Pages.

Для отдельного быстрого контура на Gemini Flash использовать `docs/GEMINI_FLASH_WORKFLOW_RU.md`.
